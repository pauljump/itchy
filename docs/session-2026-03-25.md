# Itchy Session Log — 2026-03-25

## Summary
Found and fixed a critical data leakage bug in all patching experiments. Previous 0.16 BPB results were the model copying, not predicting. Ran honest apples-to-apples comparison: byte-level is 0.18 BPB (~12%) behind token-level at matched conditions.

## The Bug

### What happened
With patch_size=12 and shift-by-1 (standard next-byte prediction), the input patch `data[0:12]` and target patch `data[1:13]` overlap by 11 bytes. The model learns to copy target bytes from the input rather than predict them. Only the last byte of each patch (1/12 = 8%) is a genuine prediction.

### How we caught it
While verifying BPB methodology before submission, analyzed the data flow:
- Input patch 0: bytes [0, 1, 2, ..., 11]
- Target patch 0: bytes [1, 2, 3, ..., 12]
- Target byte at position 0 = data[1] → sitting at input position 1. COPY.
- Target byte at position 10 = data[11] → sitting at input position 11. COPY.
- Target byte at position 11 = data[12] → NOT in input. Only genuine prediction.

### What it explained
- Why larger patches gave "better" BPB (more positions to copy)
- Why per-position decode heads helped (specialized copiers)
- Why more data didn't degrade results (copying works on any data)
- Why the numbers were suspiciously good (0.16 BPB vs baseline 1.22)

### Fix attempts
1. **Shift-by-P (patch_size)**: Input patch i predicts target patch i+1. No overlap. Got 3.97 BPB — too hard because all 12 predictions are independent.
2. **MegaByte-style causal local decoder with shift-by-1**: Still leaks. The global representation encodes the input patch, which overlaps with the target. The local decoder extracts answers from the global repr.
3. **No patching (patch_size=1)**: Standard autoregressive. Honest. Got 1.69 BPB.

## What We Learned

### Technical
1. Patching with shift-by-1 creates data leakage for patch_size > 1. This is a known issue in the MegaByte literature but easy to miss.
2. Byte-level has a ~12% BPB penalty vs token-level at matched compute. The longer sequences (2.4x) slightly outweigh the embedding efficiency gain.
3. The infrastructure gap (optimizer, multi-GPU, more data) is ~0.29 BPB — larger than the byte-vs-token gap.
4. LeakyReLU(0.5)², EMA, and the ablation methodology are all still valid independent of the patching bug.

### Process
1. Always verify evaluation methodology before claiming results
2. If a number looks too good, it probably is
3. Catching your own bug is more valuable than not having one — it proves rigor
4. The honest result (1.69 BPB) is more useful to the field than the fake one (0.16 BPB)
